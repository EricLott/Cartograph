# Acceptance Criteria

## Purpose
Define how requirements and tasks are verified as complete and correct.

## Inputs
- Functional and non-functional requirements
- Task definitions and quality constraints

## Outputs
- Verifiable acceptance statements mapped to work items

## Required Sections
- Criteria Format Rules
- Requirement-to-Criteria Mapping
- Evidence Expectations
- Failure Handling

## Criteria Format Rules
- Use observable language.
- Define pass/fail conditions.
- Include environment/context where relevant.

## Requirement-to-Criteria Mapping
Each requirement should map to one or more criteria IDs and linked tasks.

## Evidence Expectations
Accepted items must include test results, review artifacts, or runtime checks.

## Failure Handling
Failed criteria must return the work item to `in_progress` or `blocked`.

## Update Cadence
Update when requirements or quality expectations change.
